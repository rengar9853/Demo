# PHP + Redis + MySQL 高可用高并发模块化电商系统架构设计

## 1. 目标与边界

### 1.1 核心目标
- **高可用**：核心链路（浏览、下单、支付、发券）可用性目标 99.95%+。
- **高并发**：秒杀/大促场景支持流量洪峰，保障系统可降级、可限流、可恢复。
- **模块化**：领域边界清晰，支持独立演进与团队并行开发。
- **一致性与性能平衡**：交易链路采用“最终一致 + 局部强一致”策略。

### 1.2 业务模块（按你的要求）
- 用户模块
- 首页模块
- 活动模块
- 定时任务模块
- 支付模块
- 优惠券模块
- 下单模块
- 商品列表与详情模块

---

## 2. 总体架构（分层 + 领域化）

采用 **“网关层 + BFF/API层 + 领域服务层 + 数据与中间件层 + 运维治理层”**。

```text
[Client: Web/H5/App]
        |
   [CDN + WAF + SLB]
        |
 [API Gateway(Kong/Nginx)] -- JWT鉴权、限流、灰度、路由
        |
  [BFF层: PHP(FPM/Swoole)]
        |
 ------------------------------------------------------------
 | UserSvc | ProductSvc | Cart/OrderSvc | CouponSvc | PaySvc |
 | HomeSvc | ActivitySvc| SchedulerSvc  | SearchSvc | ...    |
 ------------------------------------------------------------
        |
  ------------------ 中间件与存储 ----------------------------
 | Redis Cluster | MySQL InnoDB Cluster | MQ(Kafka/RabbitMQ) |
 | Elasticsearch | Object Storage        | Config/Registry     |
  ------------------------------------------------------------
        |
 [Observability: Prometheus + Grafana + ELK + Tracing]
```

### 2.1 服务拆分建议（模块化）
- **User Service**：注册登录、用户画像、地址簿、会员等级。
- **Product Service**：SPU/SKU、价格、库存读取、商品详情聚合。
- **Order Service**：购物车、下单、订单状态机、订单查询。
- **Coupon Service**：券模板、领券、核销、回滚。
- **Payment Service**：支付下单、回调、对账、退款。
- **Activity Service**：秒杀、满减、拼团、活动规则引擎。
- **Home Service**：首页楼层、推荐位、频道聚合接口。
- **Scheduler Service**：定时任务、补偿任务、对账任务。

> 初期可采用“**单体分层 + 模块化代码**”，当流量增长后演进到微服务，避免过早拆分。

## 3. 关键技术选型

## 3.1 PHP 技术栈
- 语言：PHP 8.2+
- 框架：Laravel / Hyperf（二选一）
  - **Laravel**：生态成熟，上手快，适合常规业务。
  - **Hyperf/Swoole**：高并发、协程化更优，适合大促核心链路。
- Web：Nginx + PHP-FPM（或 Swoole 常驻进程）

## 3.2 Redis
- Redis Cluster（6 分片 + 3 主3从起步）
- 场景：缓存、分布式锁、限流、会话、热点计数、延迟队列（可选）

## 3.3 MySQL
- MySQL 8.0 InnoDB
- 结构：1 主 2 从（半同步复制）+ MHA/Orchestrator 自动切换
- 分库分表：订单、流水等高增长表按 user_id/order_id 分片

## 3.4 其他关键组件
- MQ：Kafka（高吞吐）或 RabbitMQ（可靠路由）
- 搜索：Elasticsearch（商品检索、筛选、排序）
- 任务调度：XXL-Job / 自研 Scheduler + 分布式锁
- 配置中心：Nacos / Apollo
- 服务发现：Nacos / Consul

## 4. 高可用设计

## 4.1 多副本与故障转移
- 所有应用服务至少 **3 副本**，跨可用区部署。
- 网关层双实例以上，SLB 健康检查自动摘除故障节点。
- MySQL 主从 + 自动故障切换；Redis 主从 + Cluster 自动重分片。

## 4.2 无状态服务
- PHP 应用无状态化，Session 存 Redis。
- 任何节点可水平扩展，支持弹性伸缩。

## 4.3 灰度发布与回滚
- 蓝绿/金丝雀发布。
- 配置变更与代码发布解耦。
- 关键 SQL 与缓存策略预演（影子环境）。

## 4.4 容灾策略
- 同城双机房优先（RPO < 5min，RTO < 15min）。
- 异地容灾用于极端故障（RPO < 30min）。

## 5. 高并发设计（核心）

## 5.1 流量治理
- 网关限流（令牌桶/漏桶），IP + 用户 + 接口多维限流。
- 防刷：验证码、人机校验、设备指纹、黑名单。
- 降级：非核心接口（推荐、评论）可返回兜底数据。

## 5.2 多级缓存
- 本地缓存（进程内）+ Redis + CDN 静态化。
- 商品详情页做“读扩散”：热点字段预计算。
- 缓存一致性：延时双删 + binlog 订阅修正。

## 5.3 异步化与削峰
- 下单后续流程（发券、积分、通知）通过 MQ 异步处理。
- 大促时请求入队，前端展示“排队中”。
- 库存扣减采用 Redis 预扣 + MySQL 最终确认。

## 5.4 数据库抗压
- 读写分离：读请求走从库，写请求走主库。
- 慢 SQL 治理、覆盖索引、冷热数据分层。
- 订单与流水分表，避免单表膨胀。

## 6. 交易链路设计（下单 + 支付 + 优惠券）

## 6.1 下单流程（建议）
1. 校验用户状态与收货地址。
2. 校验商品有效性与价格快照。
3. 校验优惠券可用性（预占券）。
4. Redis 预扣库存（原子 Lua）。
5. 创建订单（MySQL 事务，状态 = PENDING_PAY）。
6. 发送订单创建事件到 MQ。
7. 异步落库库存流水、营销流水。

## 6.2 支付流程
1. Pay Service 创建支付单（关联订单号）。
2. 调用第三方支付（微信/支付宝）统一下单。
3. 接收支付回调，做签名校验与幂等处理。
4. 更新订单状态为 PAID，触发履约流程。
5. 失败重试由定时任务补偿。

## 6.3 优惠券一致性
- 领券：券实例状态机（UNUSED/LOCKED/USED/EXPIRED）。
- 下单锁券：设置短期锁定超时。
- 支付成功核销；支付失败自动释放。
- 券核销与订单状态通过事件最终一致。

## 7. 模块详细设计

## 7.1 用户模块
- 登录鉴权：JWT + Refresh Token。
- 账户安全：密码加盐、设备风险识别、登录保护。
- 用户资产：余额、积分、成长值可独立子域。

## 7.2 首页模块
- 首页聚合 API（楼层、广告、推荐、活动入口）。
- 可配置化：运营后台拖拽配置楼层。
- 强缓存：首页接口可按渠道预生成。

## 7.3 活动模块
- 活动规则引擎：时间窗、人群、库存、门槛、叠加规则。
- 秒杀防超卖：库存令牌 + 下单幂等键。
- 活动数据隔离，避免拖垮主交易库。

## 7.4 定时任务模块
- 订单超时关单。
- 未支付释放库存/优惠券。
- 支付对账、退款补偿。
- 任务去重：分布式锁 + 幂等表。

## 7.5 支付模块
- 支付渠道适配层（Adapter）解耦微信/支付宝。
- 回调幂等：以 pay_no + transaction_id 去重。
- 风控校验：金额、订单状态、签名、来源 IP。

## 7.6 优惠券模块
- 券模板中心、发放规则、领取频控。
- 券可用性计算服务（支持实时与预计算）。
- 营销预算控制与核销报表。

## 7.7 下单模块
- 购物车服务化。
- 下单幂等 token（前端一次性 token）。
- 订单状态机：
  - CREATED -> PENDING_PAY -> PAID -> SHIPPED -> FINISHED
  - CREATED/PENDING_PAY -> CLOSED（超时或取消）

## 7.8 商品列表与详情模块
- 列表：ES 检索 + Redis 缓存筛选结果。
- 详情：基础信息、库存、营销价、评价聚合。
- 详情读写分离，避免强依赖交易链路。

## 8. 数据模型与分库分表建议

## 8.1 核心表
- user, user_address
- product_spu, product_sku, inventory
- cart_item
- order_main, order_item, order_payment
- coupon_template, coupon_user
- activity_rule, activity_stock

## 8.2 分片策略
- `order_main_00 ~ order_main_63`（按 user_id hash）
- `order_item_00 ~ order_item_63`
- 大字段与日志表独立库（archive/report）

---

## 9. 一致性、幂等与事务

- 跨服务事务采用 **Saga/TCC（核心场景）**。
- 事件表（Outbox Pattern）保证本地事务与消息一致。
- 幂等键：`request_id`, `order_no`, `pay_no`。
- 消息消费“至少一次”语义下，业务侧必须幂等。

---

## 10. 安全与风控

- API 鉴权：JWT + 签名 + 时间戳。
- 敏感数据：手机号/身份证脱敏与加密存储。
- 支付安全：回调白名单、重放攻击防护。
- 风控策略：异常下单频率、羊毛党识别、设备关联。

---

## 11. 监控与可观测性

- 指标：QPS、RT、错误率、库存命中率、下单成功率、支付成功率。
- 日志：结构化日志，TraceID 全链路透传。
- 告警：按 SLO 维度告警，避免“只看机器不看业务”。

---

## 12. 分阶段落地路线图

## Phase 1（0~2个月）
- 模块化单体：用户/商品/下单/支付/优惠券核心闭环。
- Redis 缓存 + MySQL 主从 + 基础监控告警。

## Phase 2（2~6个月）
- 拆分高压服务：Order、Payment、Activity 独立部署。
- 引入 MQ、ES、任务调度中心。

## Phase 3（6个月+）
- 分库分表全面上线，双机房容灾。
- 规则引擎、风控平台、数据中台能力建设。

## 13. SLA 与容量规划（示例）

- 日常峰值：3k QPS，活动峰值：20k+ QPS。
- 下单接口 P99 < 250ms（不含外部支付耗时）。
- 可用性目标：
  - 浏览链路：99.99%
  - 交易链路：99.95%

容量建议（起步）：
- PHP 应用：8~12 台（8C16G）
- Redis Cluster：6 主 6 从
- MySQL：1 主 2 从（高性能 NVMe）
- MQ：3 节点集群

## 14. 你可以直接采用的实现建议

1. **先做模块化单体**，代码按领域目录组织，避免一开始就重微服务。
2. **下单与支付先保证幂等和补偿**，比“绝对实时一致”更重要。
3. **活动和库存链路单独保护**，大促时可独立限流和降级。
4. **将首页与商品详情缓存化**，优先优化读链路成本。
5. **预留事件总线与Outbox**，为后续服务拆分做准备。
